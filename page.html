Fidelity Impact

Not all commands are dangerous.

Every command receives an FI score between 0 and 4. The score determines what stays in memory and what gets logged. It never affects routing. A separate rule-based classifier selects the appropriate execution engine, while high-impact commands are preserved longer than routine reconnaissance.

  • 0

    Read / Display

    ls, whoami, cat, pwd

  • 1

    Create / Install

    touch, mkdir, wget

  • 2

    Modify / Navigate

    chmod, rm, cd, echo > file

  • 3

    Service / Elevate

    sudo, systemctl, curl | bash, nmap

  • 4

    Impact / Delete

    passwd, rm -rf /, useradd, chmod 777

Memory retention formula

Retention = (0.7 × FI_weight)
          + (0.3 × Recency_weight)

FI_weight      = fi / 4
Recency_weight = 1 / (1 + age_min)

When the FI ≥ 2 buffer hits 10 events, the row with the lowest retention is dropped. High-FI commands survive longer — recent recon falls off first, dangerous calls stick around as context.

Rule Engine

Your rules Your trap You can control

Make HydraPoT your own. Add a YAML file to rules/ and customize responses, FI weights, routing rules, and alert thresholds instantly. No coding. No redeployment

rules/crypto_miner_trap.yaml
# match wget + .sh = fake a convincing download
- match:
    command: "wget *"
    pattern: "*.sh"
    fi_min: 1
  respond:
    template: fake_download
    exit_code: 0
    delay_ms: 420
    inject: "Saving to: '{{filename}}' [12.4K]\n██ 100%"

# high entropy pipe-to-shell = escalate + alert
- match:
    entropy_gt: 4.8
    pipe_to_shell: true
  respond:
    agent: cloud
    fi_override: 4
    alert: "siem:critical"

# anything with sudo = slow the attacker down
- match:
    command: "sudo *"
  respond:
    template: fake_sudo_prompt
    delay_ms: 1200
    loop: 3  # wrong password, 3 times

Every SSH command passes through a four-stage pipeline before a response reaches the attacker.

① Attacker connects — SSH :22
Raw paramiko session opens. Every keystroke is captured and forwarded into the pipeline. The attacker sees a convincing Ubuntu 22.04 banner.
② Honey Router — classify & route
Checks cloud escalation conditions first (entropy, obfuscation, pipe-to-shell). Then pattern-matches for Cowrie or on-device. Fallback is always on-device. Assigns an FI score 0–4.
③ FI Manager — score & remember
Stores FI ≥ 2 commands in a capped memory buffer (max 10 events). When full, drops the lowest-retention entry using: Retention = (0.7 × FI_weight) + (0.3 × Recency_weight). High-impact commands stay in context longer.
④ Agent responds
cowrie_agent — Paramiko SSH into real Cowrie container. FI 0–1.
ondevice_agent — Qwen2.5-Coder-7B locally. FI 2–3. Uses session history.
cloud_agent — GPT-4o mini via API. FI 4 + obfuscated commands.
Session logging

Every command is written to session_log.json. Commands scored FI ≥ 2 are additionally written to impactful_log.json and forwarded to the SIEM endpoint if configured.

Every command the attacker types is classified into one of five Fidelity Impact levels. The score controls routing, logging, and memory retention.

ScoreCategoryExample CommandsRouted To
0 Read / Display ls whoami cat uname pwd Cowrie agent
1 Create / Install touch mkdir wget apt-get install Cowrie agent
2 Modify / Navigate chmod rm cd echo > file On-device LLM
3 Service / Elevate sudo systemctl nmap curl | bash On-device LLM
4 Impact / Destructive passwd rm -rf / useradd chmod 777 Cloud LLM (GPT-4o mini)
Memory retention formula

When the FI ≥ 2 buffer hits 10 events, the entry with the lowest score is evicted:

Retention = (0.7 × FI_weight) + (0.3 × Recency_weight) FI_weight = fi_score / 4 Recency_weight = 1 / (1 + age_in_minutes) # Result: high-impact, recent commands survive longest. # Old recon (FI 0-1) is never stored — it doesn't compete.

The Honey Router evaluates every command through a strict priority chain. Cloud escalation is checked first — before any pattern matching — to ensure no obfuscated payload ever leaks to a weaker agent.

Decision order
PriorityConditionRoutes to
1 — Cloud check Shannon entropy > 4.8 AND length ≥ 90
OR operator count ≥ 2
OR pipe-to-shell detected (| bash, | sh)
OR hex escape sequence (\x41…)
OR base64 pattern match 		cloud_agent
2 — Cowrie patterns Basic read/display commands, file operations, package install, sudo, download tools (wget, curl without pipe) cowrie_agent
3 — On-device patterns Script execution (./, bash script.sh), service status queries (systemctl status), version queries (--version, -v) ondevice_agent
4 — Fallback Anything not matched by rules 1–3 ondevice_agent
Why cloud is checked first

A sophisticated attacker can craft commands that look simple on the surface but contain encoded payloads. Checking entropy and structural obfuscation markers before pattern matching ensures these never slip through to Cowrie (which would return a static, unhelpful response) or the on-device LLM (which may not handle extreme obfuscation reliably).

Drop a .yaml file into the rules/ directory while the honeypot is live. Rules are hot-reloaded on the next incoming command — no restart needed.

Known limitations

Cowrie filesystem

The fake filesystem inside Cowrie doesn't persist between individual commands in a session. Files "created" in one command won't appear in a subsequent ls.

bash_history sync

History sync between agents has limitations — the on-device LLM carries session context, but Cowrie's own history is not shared back.

Token estimation

Context window token counting uses a 4 chars/token approximation. Actual token counts may vary per model.

Hybrid routing

Cowrie + on-device collaboration for FI 2–3 (where Cowrie handles static parts and LLM handles dynamic) is not yet implemented.
